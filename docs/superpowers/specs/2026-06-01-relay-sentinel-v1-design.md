# RelaySentinel V1 设计文档

日期：2026-06-01
状态：待项目主人审阅

## 产品定位

RelaySentinel 是一个给中转站老板私有部署的额度监控和告警服务。第一版只解决一个核心问题：不要因为上游余额耗尽、自营号池额度不足而导致业务中断。

V1 不是 SaaS 平台，不做自动扣款，不做全量可观测性系统，也不做复杂运维平台。它应该先以“小、稳、可信”的方式跑在老板自己的服务器旁边。

## 目标用户

主要用户是中转站老板或运维者。他通常会管理：

- 需要充值、续费、找群主补额度的上游中转。
- 基于 New API、Sub2API、CLIProxyAPI 的自营中转或号池。
- 用来接收告警的群、机器人或推送渠道。

这个用户最在意的不是漂亮报表，而是在客户投诉前知道“该续费了”或“该补号了”。

## V1 范围

V1 只专注额度、余额、续费提醒。

包含：

- 手动添加上游中转：仅 New API、Sub2API。
- 手动添加自己的中转站或号池：默认 New API、Sub2API；CLIProxyAPI/CPA 作为号池巡检的高级额度来源支持。
- 明确区分“上游续费”和“号池巡检”两个模块。
- 为每个监控目标配置余额或额度阈值。
- 告警消息里带支付链接、群主联系方式、群链接或手动续费说明。
- 支持 webhook 类消息推送。
- 支持告警冷却，避免同一个问题重复刷屏。
- 记录检查历史和告警发送历史。

不包含：

- 自动支付或自动续费。
- 多租户 SaaS。
- 复杂员工权限、角色、邀请、审计系统。
- 报错监控、错误分类、失败率、延迟、性能图表。
- New API、Sub2API、CLIProxyAPI 之外的平台。
- 把个人微信/QQ 机器人作为核心能力。

## 核心模块

### 上游续费

上游续费监控的是老板向别人购买的中转或服务。

典型场景：

- 用别人的 New API 或 Sub2API 站作为上游。
- 上游提供余额查询接口。
- 上游没有在线支付，需要联系群主充值。

它回答的问题是：这个上游会不会因为余额或额度不够而断供？

每个上游续费目标需要保存：

- 名称。
- 平台类型：New API 或 Sub2API。CLIProxyAPI/CPA 不作为上游续费类型。
- 基础地址。
- 查询余额或额度所需凭证。
- 告警阈值。
- 检查间隔。
- 续费方式。
- 通知策略。

续费方式支持：

- 支付链接。
- 群链接、客服链接、群主联系方式。
- 手动说明，例如“找老王，群内 @xxx，最低充 100”。
- 无续费方式，只提醒。

告警消息必须包含当前值、阈值、目标名称和续费方式。如果某个站必须联系群主，就明确写“联系群主”，不要伪装成可以一键支付。

### 号池巡检

号池巡检监控的是老板自己的中转基础设施。

典型场景：

- New API 的渠道余额。
- Sub2API 的账号、分组、用户余额或平台额度。
- CLIProxyAPI 的账号池额度，尤其是 Codex 5 小时窗口、周额度等可被管理 API 或配套工具暴露的指标。

添加号池时，用户添加的是“自己的中转站/账号池”，不是上游。V1 默认入口先让用户选择 New API 或 Sub2API；CLIProxyAPI/CPA 只有在检测到可用 Management API 或配套额度服务时，才作为高级额度来源启用。

它回答的问题是：我自己的可用资源是不是快不够了，需要补号、续订、换号或扩容？

V1 不做复杂报错分类，但需要做“账号健康数量”巡检：每 10 分钟检测是否有账号处于错误、失效或无法正常查询额度的状态。这里的目的不是分析错误原因，而是提醒老板“有号坏了，需要处理”。告警消息只需要给出成功数、失败数、失败账号摘要和检查时间。

V1 还需要做“剩余额度可用时长”巡检：每 1.5 小时统计一次 5H 和 7D 两类额度，以过去 1 小时消耗速度为主要预测依据。当预测可用时长小于 5 小时时发送提醒。

额度巡检摘要建议包含：

- 概览：账号成功数/总数、失败数、检查时间。
- 5H 额度：总剩余百分比、预测可用时长、过去 1 小时消耗速度。
- 7D 额度：总剩余百分比、预测可用时长、过去 1 小时消耗速度。
- 预测：近 1 小时、近 3 小时、过去 1 小时消耗速度，以及 5H/7D 还能撑多久。

示例消息：

```text
概览
账号：44/45 成功，1 失败
时间：2026-05-31 18:00:33

5H额度
总剩余：98.5%
可用：4334小时
过去1小时消耗速度：0.0%/小时

7D额度
总剩余：86.7%
可用：3813小时
过去1小时消耗速度：0.0%/小时

预测
近 1 小时：0.0%/小时；近 3 小时：0.0%/小时；过去 1 小时消耗速度：0.0%/小时；5H 还能撑 4334小时；7D 还能撑 3813小时。
```

如果过去 1 小时消耗速度为 0，应显示“按当前速度暂不可耗尽”或一个足够直观的长时长，但告警判断不能因为除零而异常。近 3 小时速度用于辅助判断趋势，不作为 V1 的主要告警依据。

除上述账号健康数量和额度可用时长外，V1 不做请求失败率、泛化错误分类、延迟或性能告警。

## 平台可行性

### New API

New API 有可用的管理员渠道接口，能够支撑 V1：

- 查询渠道列表。
- 测试渠道。
- 更新全部渠道余额。
- 更新单个渠道余额。

渠道模型里包含 `balance`、`balance_updated_time`、`status`、`response_time` 等字段。V1 只优先使用余额相关字段；状态和响应时间最多作为上下文，不进入 V1 告警规则。

### Sub2API

Sub2API 对上游续费和号池巡检都有可用基础：

- 用户模型有余额。
- 账号、分组可以表达自营号池结构。
- Admin API 有用户查询、余额调整、创建并兑换充值码等能力。
- 项目内置支付和支付回调，但 RelaySentinel V1 不自动触发支付。

V1 对 Sub2API 只关注余额和额度类信号。支付相关接口可以作为未来“人工确认后续费”的基础，不能在 V1 直接用来自动扣款或自动充值。

### CLIProxyAPI

CLIProxyAPI 支持 OpenAI/Codex、Gemini、Claude、Grok 等 CLI/OAuth 多账号代理。它的 README 说明 v6.10.0 起不再内置完整用量统计，并推荐 CPA Usage Keeper、CLIProxyAPI Usage Dashboard、CPA-Manager、Quota Inspector 等配套工具。

因此 V1 对 CLIProxyAPI 要谨慎支持：

- 检测 Management API 是否可用。
- 检测是否配置了兼容的用量或额度配套服务。
- 有明确额度字段时才生成额度告警。
- 没有明确能力时显示“不可用/未配置”，不要估算或臆造额度。

这能让 CPA 支持有价值，但不依赖某个脆弱的内部假设。

## 通知渠道

V1 优先支持 webhook 类通知。

Webhook 可以理解成“消息入口链接”：RelaySentinel 发现余额不足后，向这个 URL 发一段消息，用户的群或推送软件里就会出现提醒。

建议首批支持：

- 企业微信群机器人。
- 飞书或钉钉机器人。
- Telegram Bot。
- ServerChan 或类似推送服务。
- 自定义 webhook。

QQ 和个人微信不作为 V1 一等目标。它们通常依赖非官方客户端、扫码登录、脆弱会话和较高封号风险。后续如果用户自己已经有稳定桥接服务，可以通过自定义 webhook 接入。

## 数据模型

V1 需要这些核心记录：

- 上游中转：老板向别人购买的 New API 或 Sub2API 服务。
- 自有中转站/号池：老板自己的 New API、Sub2API 部署，或带可用额度来源的 CLIProxyAPI/CPA。
- 监控目标：上游续费目标或号池巡检目标，两者在数据模型和界面入口上分开。
- 凭证：访问实例所需的加密凭证。
- 阈值规则：余额或额度阈值。
- 续费说明：支付链接、联系链接、手动说明或无。
- 通知端点：webhook 地址和消息格式配置。
- 检查结果：时间、测量值、单位、摘要。
- 告警事件：触发、冷却跳过、发送成功、发送失败、恢复。
- 额度快照：账号池在某一时刻的 5H/7D 剩余百分比、成功账号数、失败账号数。

凭证必须加密存储。能用只读凭证时，不要要求管理员级凭证。日志和界面里保存后不展示完整密钥。

## 检查流程

1. 调度器找到到期的监控目标。
2. 平台适配器按目标类型访问 New API、Sub2API 或号池侧的 CLIProxyAPI/CPA 额度来源。
3. 适配器把平台响应归一化成额度信号：
   - 当前值。
   - 单位。
   - 阈值依据。
   - 续费提示。
   - 可信度。
   - 人类可读摘要。
4. 规则引擎判断是否低于阈值。
5. 告警管理器应用冷却和恢复逻辑。
6. 通知发送器推送消息。
7. 记录检查结果和告警事件。

平台响应的复杂细节必须封装在适配器内，不要泄漏到规则引擎。

号池巡检有两个独立节奏：

- 账号健康数量巡检：每 10 分钟执行一次，只判断成功/失败数量和失败账号摘要。
- 额度可用时长巡检：每 1.5 小时执行一次，记录 5H/7D 额度快照，并用过去 1 小时消耗速度预测可用时长。

预测公式：

- 消耗速度 = 最近 1 小时剩余百分比下降值 / 1 小时。
- 可用时长 = 当前剩余百分比 / 消耗速度。
- 如果消耗速度小于等于 0，则可用时长标记为“按当前速度暂不可耗尽”。
- 近 3 小时速度只作为展示和人工判断参考。

## 告警行为

V1 告警状态：

- 正常：未低于阈值。
- 警告：低于告警阈值。
- 严重：只有用户配置了更低阈值时才启用。
- 未知：查询不到当前值。

未知不等于额度耗尽。V1 默认只在界面显示未知，不因为查询失败刷告警。后续如果用户需要“查询失败也提醒”，可以单独增加开关。

同一目标、同一规则需要告警冷却。默认建议首次触发立即提醒，持续低于阈值时每 6 小时提醒一次，恢复后可选发送恢复通知。

号池巡检的默认告警规则：

- 账号健康：失败账号数大于 0 时提醒；持续失败时进入冷却，不每 10 分钟刷屏。
- 额度可用时长：5H 或 7D 任一预测可用时长小于 5 小时时提醒。
- 额度预测不足时，告警必须同时带上 5H 和 7D 的当前剩余、消耗速度、预测可用时长，避免老板只看到“危险”却不知道该补哪类资源。

## 用户体验

第一屏应该是工作台，不是营销首页。

建议导航：

- 工作台：当前告警、低余额目标、最后检查时间。
- 上游续费：手动添加外部上游，仅 New API/Sub2API，展示余额、阈值、续费方式。
- 号池巡检：手动添加自己的中转站/号池，默认 New API/Sub2API；CLIProxyAPI/CPA 作为高级额度来源。
- 通知渠道：webhook 配置和测试发送。
- 设置：凭证、检查间隔、加密密钥提醒、备份恢复。

界面文案要贴近老板的日常语言：

- 上游续费
- 号池巡检
- 余额
- 额度
- 续费方式
- 联系群主
- 发送测试告警
- 账号成功/失败
- 5H 还能撑多久
- 7D 还能撑多久

不要在 V1 放报错监控、性能图、延迟分析。那些后续可能有用，但会稀释第一版承诺。

## 架构建议

RelaySentinel V1 做独立私有部署服务，不先做插件。

不先做插件的原因：

- New API、Sub2API、CLIProxyAPI 没有统一插件机制。
- 插件更适合监控本系统，不适合同时监控多个外部上游和自营实例。
- 每个平台写一套插件会带来版本兼容、升级和权限问题。
- 独立服务后续仍可增加轻量 agent 或插件，只在确实需要更深本地访问时再做。

推荐 V1 架构：

- 后端服务：调度器、平台适配器、规则引擎、告警管理器、通知发送器。
- 数据库：保存配置、检查结果、告警历史。
- Web UI：配置和运营查看。
- Docker Compose：方便私有部署。

V1 默认本地/私有部署。等私有部署验证用户愿意交付凭证、愿意持续使用后，再考虑 SaaS。

## 安全要求

- 凭证加密存储。
- 日志和界面隐藏敏感信息。
- 提供连接测试，但不展示敏感响应。
- V1 不做广泛管理员写操作。
- V1 不执行支付、充值、余额修改、账号删除、自动修复。
- 备份/导出要明确提醒：配置里可能包含加密后的敏感凭证。

## 测试策略

第一版实现计划应覆盖：

- 阈值判断和告警冷却单元测试。
- New API、Sub2API、CLIProxyAPI 适配器契约测试。
- 使用假 webhook 的通知发送测试。
- 调度到告警的集成测试。
- 每个平台至少一个本地 mock 实例形态的手动冒烟测试。
- 额度可用时长预测测试，覆盖 1 小时消耗、3 小时趋势、0 消耗速度和小于 5 小时告警。
- 账号健康数量巡检测试，覆盖 44/45 成功、1 失败这类摘要输出。

最高风险是凭证处理和重复告警逻辑。它们应该先有测试，再去打磨界面细节。

## 产品风险

最大风险是凭证信任。私有部署可以降低这个风险。

第二风险是平台 API 漂移。适配器要容忍版本差异，并明确提示“不支持/未配置”，不能默默误判。

CLIProxyAPI 的额度能力可能因安装方式不同而不同。除非检测到兼容 Management API 或配套额度服务，否则不能承诺完整 CPA 额度巡检。

通知可靠性比界面好看更重要。漏报一次，比表格丑十倍更伤信任。

## 里程碑

### 里程碑 1：最小闭环

- 创建服务骨架。
- 手动添加一个 New API 上游。
- 加密保存凭证。
- 跑通一个 New API 余额检查。
- 触发一次 webhook 告警。
- 记录检查和告警历史。

### 里程碑 2：V1 平台覆盖

- 增加 Sub2API 适配器。
- 增加 CLIProxyAPI 能力探测和额度能力处理。
- 增加续费方式字段。
- 增加“添加上游”和“添加自己的中转站/号池”两条手动入口。
- 增加 10 分钟账号健康数量巡检。
- 增加 1.5 小时 5H/7D 额度可用时长巡检。

### 里程碑 3：可用运维体验

- 工作台。
- 告警冷却和恢复提醒。
- 通知渠道测试按钮。
- Docker Compose 私有部署。
- 备份和恢复说明。

## 奥卡姆剃刀审视

这版设计只保留最能产生价值的链路：查额度、比阈值、发提醒、带续费信息。

暂时砍掉自动支付、多租户、复杂权限、报错监控、性能监控、个人微信/QQ 机器人，是为了避免第一版从“能赚钱的小工具”变成“做不完的大平台”。
