总览架构

RelaySentinel 的本质是一个“中转和号池哨兵”：持续知道哪里快断供，提前通知老板处理。

后端测试39 passed
前端测试6 passed
核心平台New API / Sub2API
V1 边界不自动付款

前端现在是什么

当前 `web/` 是 React + TypeScript + Vite 的移动端 PWA 预览。它展示老板每天打开手机时想看的东西：未处理告警、上游余额、号池额度、通知渠道。

前端还缺什么

目前主要用 `web/src/data/mockData.ts`，还没有完整接入真实后端 API。下一步要补 API client、加载/错误状态、表单提交和告警操作回写。

关键文件

  • web/src/App.tsx：PWA 页面和交互
  • web/src/domain/quota.ts：前端额度计算
  • web/src/data/mockData.ts：演示数据
  • web/public/manifest.webmanifest：PWA 安装信息

我会优先改什么

先把首页、上游列表、号池列表、告警处理接真实 API。不要先做复杂设置页；能让老板收到告警并确认处理，才是第一收益点。

入口层

relay_sentinel/app.py 创建 FastAPI app，注册所有 V1 API。relay_sentinel/asgi.py 读取环境变量，给 uvicorn 部署用。

存储层

relay_sentinel/storage.py 直接使用 SQLite。优点是少依赖、容易部署；缺点是未来多人/多实例要迁移到 PostgreSQL 或做更严格锁策略。

适配器层

adapters/new_api.py 读 New API 通道余额，adapters/sub2api.py 登录 Sub2API 并读取余额。Cloudflare 阻断会单独报错，不混成密码错误。

领域层

domain/quota.py 算额度还能撑多久；alerts.py 做冷却；pool_health.py 汇总账号健康；scheduler.py 只做 due check 选择。

已经做了

  • API 响应不返回明文 password/token。
  • SQLite 不保存原始 `"password"` / `"token"` JSON 字段名。
  • V1 不提供充值、付款、改余额接口。
  • 错误响应不会回显提交的 secret。

还不够生产

  • 凭证封存是测试级替身，不是 KMS。
  • 没有登录权限，不能直接公网裸露。
  • webhook 真实投递缺少重试、签名和审计。
  • SQLite 单机可以，多实例不够。

关键风险

如果这个项目要真的管钱，第一风险不是页面不好看，而是凭证和告警可靠性。密钥管理、访问控制、投递审计必须比花哨 dashboard 更早做。

生产底线

放到公网前至少要加反代 HTTPS、访问鉴权、强随机 secret key、数据库备份、日志脱敏，以及 webhook 投递失败可追踪。

本地启动

uvicorn relay_sentinel.asgi:app --host 0.0.0.0 --port 8000 --reload

单机部署

README 推荐 systemd + SQLite + Nginx/Caddy 反代。V1 更适合先单机跑稳，不急着做 Kubernetes。

验证命令

  • pytest tests/backend -q
  • cd web && npm test
  • cd web && npm run build

还缺运维件

健康检查端点、结构化日志、指标、数据库备份、真实 scheduler、部署脚本和 release checklist。

1
真实定时巡检老板不可能每天手动点检查。定时查上游余额、号池健康、5H/7D 额度，是这个项目的核心价值。
2
通知投递审计提醒没送到，比没提醒更危险。要记录每次 webhook 的成功、失败、响应码和重试。
3
前端接真实 API当前 PWA 已有体验，但还用 mock 数据。要让它成为老板每天打开的真实工作台。
4
生产级安全凭证、webhook、后台服务都和钱相关。上线前必须补鉴权、密钥管理、备份和日志脱敏。
5
再考虑多租户先别急着 SaaS 化。先服务一个老板，把断供预警做准，再谈更多客户。

老板 / 值班人
